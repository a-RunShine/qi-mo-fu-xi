## 三、各章知识点大纲（基于搜集资料确认，已修正错误）

### 第一章 软件与软件工程

**核心知识点：**
1. **软件危机**
   - 表现：成本超支、进度拖延、质量低下、维护困难
   - 原因：客观（软件规模/复杂度激增）；主观（缺乏管理+方法）
   - 消除途径：技术措施 + 组织管理措施 + 正确认识 + 总结经验
2. **软件工程**
   - 定义：1968年NATO会议提出，工程化方法开发维护软件
   - 三要素：方法、工具、过程
   - 本质特性：关注软件质量
3. **软件生命周期**
   - 三时期：定义时期（问题定义+可行性研究+需求分析）、开发时期（总体设计+详细设计+编码+测试）、维护时期
   - 各阶段输出文档（需求规格说明书、设计说明书等）
4. **软件过程模型**（高频考点，出现频率32%）
   - 瀑布模型：线性、文档驱动、需求明确
   - 快速原型模型：先建原型确认需求
   - 增量模型：分批交付、非整体开发
   - 螺旋模型：瀑布+快速原型+风险分析（Boehm提出）
   - 喷泉模型：面向对象、迭代无缝
   - RUP：用例驱动、以体系结构为中心、增量迭代
   - 敏捷过程：快速响应变化、Scrum/XP
5. **敏捷宣言4条价值观**（必考点，需完整列出）
   - 个体与互动 > 流程与工具
   - 工作的软件 > 详尽的文档
   - 客户合作 > 合同谈判
   - 响应变化 > 遵循计划

**选择题坑点方向：**
- 螺旋模型 = 瀑布 + 快速原型 + 风险分析（不是+增量模型！）
- 瀑布模型最大缺点不是"阶段划分清晰"，而是"缺乏灵活性、无法应对需求变化"
- 敏捷宣言价值观的反向设问（"以下哪项不是敏捷宣言的价值观"或"以下哪项的优先级表述错误"）
- 增量模型 vs 螺旋模型的迭代层级区别

### 第二章 可行性研究与计划

**核心知识点：**
1. **可行性研究实质**：简化、压缩的系统分析与设计
2. **三类可行性**
   - 技术可行性：能否实现
   - 经济可行性：成本效益
   - 操作可行性：运行环境是否可行
   - （部分教材加：法律/社会可行性）
3. **系统流程图**：描述物理系统，掌握符号
4. **数据流图（DFD）**
   - 四种基本符号：源点/终点（正方形）、处理（圆角矩形）、数据存储（开口矩形）、数据流（箭头）
   - 分层：顶层→0层→1层...
   - **父图与子图的平衡原则**（高频考点）：子图的输入输出数据流必须与父图中对应处理的输入输出一致
5. **数据字典（DD）**
   - 四类元素：数据流、数据项、数据存储、处理
   - 与DFD互补：DFD+DD共同构成系统规格说明
6. **成本效益分析**
   - 货币时间价值计算（贴现/净现值）
   - 投资回收期、纯收入
7. **成本估算**
   - 代码行估算（LOC）
   - 功能点估算（FP）
   - COCOMO模型（基本/中间/详细，精度递增）

**选择题坑点方向：**
- 可行性研究的三个维度场景判断（容易把"操作可行性"和"技术可行性"混淆）
- DFD符号的图形对应
- DFD父图子图平衡原则的判断
- 数据字典与DFD的关系（不是替代，是互补）
- COCOMO三种模型的精度层级

### 第三章 软件需求分析

**核心知识点：**
1. **需求分析任务**：建立系统逻辑模型
2. **需求获取方法**：访谈、问卷、情景分析、快速原型
3. **需求分类**
   - 功能需求：系统必须做什么
   - 非功能需求：性能、可靠性、安全性等
   - 领域需求：来自应用领域
4. **三种模型**
   - 数据模型（ER图）
   - 功能模型（DFD）
   - 行为模型（状态转换图）
5. **结构化分析（SA）**
   - DFD、数据字典、实体-联系图、状态转换图
6. **需求规格说明书（SRS）**：需求分析的输出文档
7. **需求验证**
   - 一致性、现实性、完整性、有效性
8. **UML基础**（面向对象方法）
   - 用例图：参与者、用例、关系（包含、扩展、泛化）
   - 类图：类、属性、方法、关联

**选择题坑点方向：**
- 功能需求 vs 非功能需求 vs 设计约束的场景判断
- 用例图的"包含"（include）vs"扩展"（extend）关系
- 三种模型分别用什么图表示
- 需求验证的四个维度

### 第四章 软件设计（总体设计+详细设计）

**核心知识点：**
1. **设计原理**
   - 模块化、抽象、信息隐藏、局部化
2. **模块独立性**（核心考点）
   - **内聚**（模块内，越高越好）：偶然→逻辑→时间→过程→通信→顺序→功能
   - **耦合**（模块间，越低越好）：内容→公共→外部→控制→特征→数据→非直接
3. **启发规则**
   - 改进结构、模块规模适中、扇出3-4、扇入越大越好
   - 控制域应包含作用域
4. **图形工具**
   - 层次图、HIPO图、结构图（掌握结构图符号：模块、调用、数据、控制信号）
5. **面向数据流设计**
   - 变换流、事务流
6. **详细设计**
   - 结构程序设计（三种基本控制结构：顺序/选择/循环）
   - 判定表、判定树、PDL（过程设计语言）
   - 程序流程图符号（与McCabe计算相关，需会画/读）
   - **McCabe环形复杂度**：V(G) = E - N + 2（边数-节点数+2）

**选择题坑点方向：**
- 内聚7种从低到高排序（功能最强，偶然最弱）
- 耦合7种从高到低排序（非直接最低，内容最高）
- "数据耦合是高耦合"（错！数据耦合是低耦合，是希望的）
- 控制耦合传递的是控制信息（flag），不是数据
- McCabe复杂度计算（需会读流程图数边和节点）
- 扇出 vs 扇入的方向

### 第五章 软件测试与维护 ★★★（用户明确重点，题量25-30题）

**核心知识点：**
1. **软件测试目标**
   - 发现错误（不是为了证明程序正确）
   - 好测试 = 高发现概率 + 高覆盖
   - 由独立第三方测试效果更好
2. **黑盒测试（功能测试/基于规格说明的测试）**
   - 依据：需求规格说明书
   - 不考虑内部结构
   - 方法：
     - 等价类划分（有效+无效等价类，划分规则）
     - 边界值分析（上点、离点、内点，是等价类划分的补充）
     - 因果图法
     - 判定表驱动
     - 错误推测法
     - 场景法
   - 适用：系统测试、验收测试
3. **白盒测试（结构测试/基于结构的测试）**
   - 依据：程序流程图、源代码
   - 逻辑覆盖法：
     - 语句覆盖（每条语句至少执行一次）— 最弱
     - 判定覆盖/分支覆盖（每个判定的真假分支至少执行一次）
     - 条件覆盖（每个条件的真假至少执行一次）
     - 判定/条件覆盖
     - 条件组合覆盖
     - 路径覆盖（所有可能路径至少执行一次）— 最强
   - 基本路径测试法：McCabe复杂度，独立路径数 = V(G)
   - 适用：单元测试
4. **覆盖率关系（重要！偏序关系，非完全线性）**
   - 语句覆盖 ⊂ 判定覆盖（判定覆盖一定包含语句覆盖）
   - 条件覆盖 与 判定覆盖 **不可比**（互不包含，常考坑点！）
   - 判定/条件覆盖 ⊇ 判定覆盖 且 ⊇ 条件覆盖
   - 条件组合覆盖 ⊇ 判定/条件覆盖
   - 路径覆盖 ⊇ 条件组合覆盖（在有限路径前提下）
   - 推荐记忆链：语句 ⊂ 判定 ⊂ 判定/条件 ⊂ 条件组合 ⊂ 路径（条件覆盖独立于该链，仅与判定不可比）
   - 路径覆盖最强但也不能保证发现所有错误（含循环时无法穷举路径，且路径覆盖不一定包含条件组合覆盖）
5. **测试步骤**
   - 单元测试（模块测试）：白盒为主
   - 集成测试：自顶向下 / 自底向上 / 三明治（混合）
     - **自顶向下**需要桩模块（stub，模拟下层模块）
     - **自底向上**需要驱动模块（driver，模拟上层模块）
   - 系统测试：黑盒为主
   - 验收测试：α测试、β测试
6. **α测试与β测试**（以张海藩教材为准）
   - α测试：在**开发者的环境**中进行，由**用户**进行测试，开发者通常**不在场**
   - β测试：在**用户的实际运行环境**中进行，开发者通常**不在场**
   - 核心区分点：**测试地点**（开发环境 vs 用户实际环境）和**测试阶段**（α在前、β在后）
7. **回归测试**：修改后重新测试，防止引入新错误
8. **调试（Debug）方法**：强行排错、回溯法、原因排除法
9. **软件维护**
   - 四种类型：改正性、适应性、完善性（占比50%以上，最大）、预防性
   - 维护成本：软件维护成本远高于开发成本
   - 软件可维护性：可理解性、可测试性、可修改性

**选择题坑点方向（重点设计）：**
- 白盒看不到"用户想要的功能错误"，只管代码逻辑
- 语句覆盖最弱，路径覆盖最强，但路径覆盖也不能保证完全正确
- 条件覆盖 ≠ 判定覆盖（两者不可比，互不包含）
- 判定覆盖一定包含语句覆盖
- α测试 vs β测试的区分点是**测试地点**（不是开发者是否在场）
- 集成测试自顶向下需桩模块，自底向上需驱动模块
- 维护类型占比（完善性维护占50%以上）
- 测试目标不是"证明程序正确"，而是"发现错误"
- 边界值分析是等价类划分的补充，不是替代

### 第六章 软件项目管理（面向对象仅作背景，不作为主要考点）

**核心知识点：**
1. **进度管理**
   - 甘特图：直观显示进度
   - PERT图/CPM（关键路径法）：计算最短工期
   - **关键路径**：耗时最长的路径，决定项目最短工期（不是最短路径！）
   - PERT期望时间公式：(乐观时间 + 4×最可能时间 + 悲观时间) / 6
   - PERT标准差公式：(悲观时间 - 乐观时间) / 6
2. **风险管理**
   - 四步：识别→估计→驾驭→监控
   - 应对策略：规避、转移、减轻、接受
3. **软件质量保证（SQA）**
   - 技术审查、程序正确性证明、走查（Walkthrough）、审查（Inspection）
4. **软件配置管理（SCM）**
   - 配置项、基线、版本控制、变更控制、配置审计
5. **CMM（软件过程能力成熟度模型）五级**（以张海藩教材为准，采用CMM叫法）
   - 初始级（1级）：过程不可预测
   - 可重复级（2级）：项目级纪律
   - 已定义级（3级）：组织标准过程
   - 已管理级（4级）：用定量指标度量过程（注：CMMI中叫"量化管理级"）
   - 优化级（5级）：持续改进
6. **面向对象基础**（背景铺垫，非主要考点）
   - 类、对象、继承、封装、多态、消息
   - UML图分类

**选择题坑点方向：**
- 关键路径 = 最长路径（不是最短！）
- CMM等级特征的反向设问（给出特征判断等级）
- CMM第4级"已管理级" vs CMMI"量化管理级"的名称差异（教材差异提示）
- 风险应对策略的场景判断
- PERT期望时间与标准差的计算

## 四、AI讲解设计规范

### 4.1 文档结构（每章3000-5000字）

```

- 一句话概括本章核心
- 本章在考试中的地位（高频考点提示）

### 核心概念
（定义+解释）
### 为什么重要
（原理深入）
### 易混辨析
（对比易混淆概念）
### 记忆技巧
（口诀/类比/图表）

## 二、{知识点2}
...（每个知识点按顺序编号，重复此结构）

## N、{最后一个知识点}

- 常见题型

```

注：知识点部分用"## 一、二、三...N"顺序编号（按实际知识点数量），后续固定栏目（考点聚焦/记忆口诀总览/本章小结）不带数字序号，避免与知识点序号冲突。对教材有差异的考点，在"易混辨析"下设"教材差异提示"小栏目。

### 4.2 写作要求
1. **讲穿不浮于表面**：每个概念都要解释"为什么"，不只给定义
2. **生活化类比**：用现实场景类比抽象概念（如"内聚像团队协作，耦合像部门扯皮"）
3. **记忆口诀**：每章提供3-5个记忆口诀，如"内聚七级：偶逻时过通顺功"（偶然→逻辑→时间→过程→通信→顺序→功能）
4. **易混辨析**：用对比表格区分易混淆概念
5. **考点聚焦**：明确标注哪些是高频考点
6. **教材差异提示**：对α/β测试、CMM第4级名称、逻辑覆盖排序等差异考点，说明"张海藩版说法 vs 其他标准说法"

---

## 五、选择题设计规范

### 5.1 题型分布
- 第五章（重点）：25-30题
- 其他章节：15-20题
- 题型占比：
  - 概念辨析题（40%）：易混淆概念区分
  - 场景应用题（30%）：给定场景判断方法/模型/工具
  - 排序/分类题（20%）：内聚排序、CMM等级
  - 计算/推理题（10%）：关键路径、McCabe复杂度、覆盖率、PERT

### 5.2 质量要求
1. **有坑**：每个错误选项都要有迷惑性，不能一眼排除
2. **绕弯子**：不直接问定义，而是通过场景/对比/反向设问
3. **解析详尽**：不仅给答案，要解释每个选项为什么对/错
4. **标注坑点**：每题标注【坑点】说明常见错误

### 5.3 题目与解析解耦（两文件）

每章选择题拆分为两个文件：
- **`第X章_XXX_选择题.md`**：只含题目、选项、答案（纯净，便于做题）
- **`第X章_XXX_选择题解析.md`**：含每题的详细解析和坑点说明

**题目文件格式：**

```
### 第X题（单选）
题目内容...

A. 选项A
B. 选项B
C. 选项C
D. 选项D

**答案**：B
```

**解析文件格式：**

```
### 第X题
**答案**：B

**解析**：
正确选项B的原因...
错误选项A/C/D各自的错误点...

【坑点】常见错误是选C，因为...；正确答案应该是B，因为...
```

注：答案与坑点中的"常见错误选项"使用不同字母区分，避免混淆。解析独立成文件，便于做完题后对照。
