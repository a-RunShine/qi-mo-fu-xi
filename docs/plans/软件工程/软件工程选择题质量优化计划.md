# 软件工程选择题质量优化计划

> 已经过 explore 审查并据此优化。审查关键结论已融入下文。

## 一、总体策略

- **原地改造为主，接受合并导致的题数减少**：保持每章主体结构，把太浅的题改写为有深度的场景题；合并题后题数自然减少，文件头"共X题"标注同步更新，不补新题
- **基础题保留 20-25%**：每章只留纯记忆必考点（★★★及核心概念），其余全部深化
- **深度标准**（参照 `数据库/复习题/选择题/第四章_数据库安全性_选择题.md`，满足至少2条）：
  - 场景化题干（具体操作序列/角色/代码/数据/网络图）
  - 选项微差，需推理才能区分
  - 合理陷阱（看似正确实则错误，且错得有道理——对应常见误解或相邻概念）
  - 不能仅靠背诵定义答出，需推理或应用
- **学科适配**（软工与数据库的本质差异）：
  - 数据库深度题是"硬机制追踪"（CASCADE有确定的权限传播链和唯一结果）
  - 软工深度题多为"定性判断"（判耦合类型、判过程模型），没有唯一机制路径
  - 因此软工的"拆解路径"不是追踪授权链，而是**拆判断依据**（如判耦合：看传什么→是数据还是控制→决定类型）
  - 软工的"选项微差"更难设计，因为答案是定性的，干扰项更易"明显反着说"——必须让每个干扰项错得有道理，对应一个具体误解
- **无 PDF 同步**：软工选择题无 PDF（区别于数据库），改造只动 .md 文件

## 二、总体诊断

6 章共 112 题（一18 / 二16 / 三16 / 四18 / 五28 / 六16）。整体偏基础，对照数据库第四章样本（10 题中 6-7 题有场景+机制陷阱+选项微差），软工 112 题里真正"有深度"的只有 2-3 题（集中在第五章覆盖关系），其余全是直问直答或"一句话特征套口诀"。

**改进优先级排序（高→低）：**
1. 第四章 软件设计 — 内聚耦合判断本可给伪代码却只给一句话特征
2. 第五章 测试维护 — 28题仅2-3道深度题，逻辑覆盖本可给代码判级别却考定义
3. 第六章 项目管理 — 关键路径本可给AOE图算却全考概念；且第4题有数据bug
4. 第一章 软件工程 — 过程模型选用场景太浅
5. 第二章 可行性研究 — DFD平衡、经济计算可深化
6. 第三章 需求分析 — 需求分类判断方向对但太简单

**主要差距：**
1. 缺场景化（最大差距）：数据库样本几乎每题有操作序列/角色/密级场景；软工112题里约15题带场景，且多为"传flag""用户抵触"这种单一特征
2. 选项无区分度：软工正确选项常明显，干扰项"明显错误"
3. 靠记忆即可：多数题套一句口诀就能答，无需推理

## 三、执行前置步骤（动手前必须完成）

### 3.1 教材口径确认

通读各章解析里已标注"以张海藩教材为准"的题，汇总成口径清单。已知分歧点至少有：
- **耦合最低类型**：非直接耦合 vs 数据耦合（本计划采用"非直接耦合最低"，解析显式标注教材差异）
- **α测试定义**：地点（开发环境）与主体（用户）——以张海藩教材为准
- **CMM 第4级名称**：已管理级 vs 定量管理级（以张海藩"已管理级"为准）
- **过程模型权衡题答案**：多约束场景下教材推荐哪个模型，需确认

改造时遇分歧，解析里显式标注"以XX教材为准，部分教材记为YY"。

### 3.2 样板章机制

**第四章作为样板章先行改造**，改完按本计划 5.6 检查清单逐题验收通过后，再推广到其他章。避免同样问题重复六次。

### 3.3 图形文本化规范

软工选择题纯 .md（无 PDF），图形只能用文本。统一采用以下表达方式：

**模块结构图**（第四章扇出扇入题用）——缩进列表表示层级：
- 模块A（扇出7）调用 B/C/D/E/F/G/H
- B 同时被模块X调用（B扇入2）

**流程图**（第四章McCabe题用）——编号节点+边列表：
- 节点1（开始）→[无条件] 节点2
- 节点2 →[条件a] 节点3
- 节点5 →[条件b] 节点2（循环回）

若文本表达可读性差，退回半数值化描述（直接给"8节点10边"），不硬画ASCII框图。

**AOE网络图**（第六章关键路径题用）——任务表+依赖关系文字描述：
- 任务A：乐观3/最可能6/悲观15，紧前无
- 任务B：乐观4/最可能5/悲观13，紧前A
- 依赖：A→B, A→C, B→D, C→D（D为结束节点）

不画箭头图，用任务表+依赖关系文字，既保留场景感又文本可读。

**用例图**（第三章用例关系题用）——"参与者—用例—关系"三元组列表：
- 参与者：客户
- "下单" include "登录"（下单必含登录）
- "下单" extend "使用优惠券"（下单时可选使用优惠券）
- "下单" 泛化 "货到付款下单"（子用例继承父用例）

### 3.4 伪代码格式规范

第四章多处用伪代码，统一规范：
- 类C风格，≤10行
- 函数名/变量名用中文有语义的命名（如 `计算平均成绩` 而非 `fn1`）
- 控制流用 if/else/for/while
- 示例：`A(flag, score){ if(flag==1) return 平均(score); else return 最大值(score); }`

### 3.5 题干长度软上限

- 单题题干 ≤150 字
- 合并题题干 ≤200 字
- 超了就砍：5条工单砍到3条，5个用例砍到3个

### 3.6 合并题题号处理规则

- 合并后**全章重新连号 1-N**，不留空缺题号
- 文件头注明"本卷共N题，已合并原X/Y题"
- 解析文件与题目文件题号一一对应
- 答案速查表按新题号重排
- 执行顺序：先决定合并→重排全章题号→更新答案速查表→再逐题改内容

### 第一章 软件与软件工程（18题，无合并，保持18题）

**保留基础题（4题）：**

**改造题（预期答案+陷阱）：**

- **第1题** 危机原因vs表现
  - 场景：某项目预算2000万实际花了5000万，工期拖延一年，交付后用户拒绝签收。问以下哪项是危机的"原因"而非"表现"。
  - 预期答案：C.软件的逻辑复杂度不可见，难以估算工作量
  - 陷阱：A.预算超支 / B.工期拖延 / D.用户拒收——这三项都是危机的"表现"（结果），C才是"原因"（根源）。考生易把表现当原因。

- **第3题** 软件本质特性
  - 场景：同样是"造一座桥"和"开发一个APP"，为什么APP更容易进度失控？问软件区别于其他工程的最本质特性。
  - 预期答案：B.软件是逻辑产品，复杂度不可见且不随制造而降低
  - 陷阱：A.软件成本高 / C.软件需要团队 / D.软件用计算机运行——这些都不是"区别于其他工程"的本质（造桥也花钱也要团队也用工具），B才是软件独有的逻辑本质。

- **第5题** 瀑布模型特性导致返工
  - 场景：某瀑布项目在编码阶段才发现需求理解有误，导致需求文档、设计文档全部重写，返工巨大。问瀑布的哪个特性导致了这一后果。
  - 预期答案：C.文档驱动+严格顺序，前一阶段未完成不能进入下一阶段
  - 陷阱：A.阶段清晰（这是优点不是返工原因）/ B.强调文档（片面，单独文档不会导致返工，是文档驱动+严格顺序共同导致）/ D.有测试阶段（无关）

- **第7题** 多约束权衡（★★★深化）
  - 场景：某军工项目风险很高，但需求已较明确（军方提供了详细规格说明），且要求6个月内必须交付第一版可用系统。问最合适的模型。
  - 预期答案：B.增量模型（需求明确可不靠原型，但6个月出第一版需要增量分批交付，风险高可每增量做评审）
  - 陷阱：A.螺旋模型（风险分析最合适，但忽略"需求已明确+6个月出第一版"，螺旋每轮风险分析耗时，难保6个月）/ C.瀑布模型（需求明确看似合适，但6个月出第一版与瀑布的"全部完成才交付"矛盾）/ D.喷泉模型（面向对象，与场景无关）
  - 说明：权衡题答案因教材而异，以张海藩教材"增量模型适合分批交付"为准，解析显式标注权衡逻辑。

- **第8题** 增量vs螺旋判断
  - 场景：某项目需求模糊、技术风险高、客户希望逐步看到成果。给四个判断，问哪项正确。
  - 预期答案：B.螺旋更适合，因为风险高需每轮风险分析
  - 陷阱：A.增量也能做风险分析所以更合适（增量无系统化风险分析机制，这是螺旋特有）/ C.两者都行无区别（忽视了风险分析是螺旋核心）/ D.应选瀑布因为客户要看到成果（与需求模糊矛盾）

- **第11题** 敏捷vs瀑布vs螺旋权衡
  - 场景：6人小团队开发内部工具，需求变化快，但公司要求有完整设计文档备审计。
  - 预期答案：C.难以完全用敏捷，需在敏捷基础上补充文档交付，或选RUP
  - 陷阱：A.用敏捷响应快（忽略文档审计要求）/ B.用瀑布保证文档（忽略需求变化快）/ D.用螺旋做风险分析（场景无高风险，过度）

- **第18题** 瀑布vs增量权衡
  - 场景：某OA系统需求明确，但后期可能要对接新收购公司的异构系统。
  - 预期答案：B.增量模型，先交付核心OA，后续增量对接异构系统
  - 陷阱：A.瀑布模型（需求明确看似合适，但对接需求不确定，瀑布难应对后续变化）/ C.螺旋（无高风险，过度）/ D.喷泉（非OO场景）

  - 预期答案：C.100×(1.05)³≈115.8万
  - 陷阱：A.100万（未算利息）/ B.115万（算成单利100×1.15）/ D.130万（算错）

- **第16题** DFD平衡判断

  - 陷阱：A.子图1违反（子图1输入输出与父图一致，没违反）/ C.都违反 / D.都没违反（子图2输入不一致）

## 五、解析生产计划

### 5.1 现状诊断

对比数据库第四章解析（参考样本）与软工现有解析，差距显著：

| 维度 | 数据库参考样本 | 软工现状 |
|---|---|---|
| 开篇 | 一句话说穿考点本质 | 直接进"答案→解析"机械结构 |
| 拆解 | 拆机制路径 | 复述选项，不拆内在逻辑 |
| 选项分析 | 逐项说为什么对/错，揭示强干扰项错在哪 | 只标"X错" |
| 微差揭示 | 点明一字之差 | 不区分相近选项 |
| 收尾点睛 | 引用块给比喻/口诀，恍然大悟 | 【坑点】多是背诵口诀 |
| 本质洞察 | 揭示"为什么是这样" | 停留在"是什么"的复述 |

### 5.2 解析质量标准

**深度题用六要素，基础题用两要素。**

**深度题六要素：**

1. **点明本质（开篇一句话）**：开头一句话说穿这题在考什么机制。不绕弯，直接点破。
   - 正例："这题考的是CASCADE级联回收的范围——回收一条授权路径，不影响其他独立路径。"
   - 反例："本题考查数据库安全性。"（空话）

2. **拆解路径（还原机制过程）**：把题目场景的内在逻辑一步步拆开。软工的"拆解路径"是拆判断依据（如判耦合：看传什么→是数据还是控制→决定类型），不是追踪授权链。

3. **逐项剖析（说清每个选项的对错理由）**：每个选项不只标对错，要说明为什么对、为什么错、错的干扰项对应什么常见误解。强干扰项用"（强干扰项，与X的差别在...）"标注，普通错项用"（错，对应XX误解）"标注。

4. **揭示微差（仅强干扰项适用）**：若题目有强干扰项，必须点明正确项与干扰项的差别在哪一个词、哪个机制点。无强干扰项的题不必硬凑。

5. **点睛比喻（收尾恍然大悟）**：用引用块给一个比喻、口诀或一句话本质，让人读完"哦——原来如此"。比喻要精准对应机制，不能为比喻而比喻。

6. **延伸洞察（深度题必须有）**：对反直觉或易错的点，补充"为什么是这样"的底层逻辑，让理解超越这道题本身。
   - 正例："补充：MAC还有对称的向上写规则——防止高密级数据流向低密级主体。"（揭示设计意图）

**基础题两要素（本质+点睛）：**

基础题解析简洁，只含：点明考点 + 引用块点睛口诀/本质一句话。逐项说明仅在选项有混淆价值时才加（如非直接耦合vs数据耦合的混淆值得逐项，内聚最高就不值得）。

### 5.3 解析格式模板

**深度题模板：**

```
### 第X题

**答案：X**

**解析：**

[要素一：一句话点明本质]

[要素二：拆解路径，还原机制过程]

[要素三+四：逐项剖析，强干扰项揭示微差]
- A：...（错，对应XX误解）
- B：...（正确，因为XX）
- C：...（强干扰项，与B的差别在XX）
- D：...（错）

> [要素五：点睛比喻/口诀]
>
> [要素六：延伸洞察，深度题必须有]

---
```

**基础题模板：**

```
### 第X题

**答案：X**

**解析：**

[一句话点明考点]

> [口诀或本质一句话]

```

### 5.4 生产执行要求

1. **与题目改造同步**：每章改完题目后，立即重写该章解析文件，不留到后面批量补
2. **逐题重写**：不沿用旧解析，即使题目未改的基础题，解析也要按新标准重写
3. **答案一致性三层核对**：题目答案 → 解析答案 → 答案速查表，三处必须一致
4. **格式统一**：沿用数据库第四章解析的格式
5. **保留答案速查表**：解析文件末尾保留答案速查表，与题目文件答案对齐
6. **按章分批提交**：每章改完即提交，不要六章攒一起

1. （已完成）写本计划到 `plans/软件工程选择题质量优化计划.md`
2. （已完成）explore 审查计划并据此优化
3. 改造第四章（样板章）→ 按 5.5 检查清单逐题验收 → 通过后推广
4. 按优先级改造：第五章 → 第六章 → 第一章 → 第二章 → 第三章
5. 每章改造流程：决定合并→重排题号→改题目文件→同步重写解析文件→三层核对答案
6. 全部完成后通读检查：题号连续无空缺、答案三处一致、合并题题干可读、格式一致

### 6.2 验证标准

- 每章改完后核对：基础题占比约20-25%、深度题满足"至少2条标准"、答案与解析一致
- 第六章第4题确认 σ=(13-4)/6=1.5 精确匹配选项B
- 第六章第2题确认各任务期望值、关键路径计算正确
- 合并题需确认原题号不冲突、解析文件题数对齐、题干≤200字
- 格式与现有文件保持一致（题号、选项、答案位置、★★★标记等）
- 解析达到六要素（深度题）/两要素（基础题）标准，逐题通过5.5检查清单
