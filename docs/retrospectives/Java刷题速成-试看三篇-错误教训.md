## 一、文档内容错误(7 个高优先级,均已修复)

### 1.1 【严重】00 总览表内部矛盾

- **表现**:`00_总览与速成路线.md` 第 47 行 `printf → String.format`,第 61 行 `printf("%d") → StringBuilder.append`,两条对同一 C++ 概念的映射互相矛盾
- **根因**:subagent 写对照表时按行生成,没检查"行与行之间映射一致性"
- **修复**:删除错行,改用 `ostringstream → StringBuilder`(C++ "输出到字符串" 语义对应更准)
- **预防**:生成对照表后,**必须**抽 3-5 个 C++ 概念(常见如 `printf` / `string` / `sort`),查它们在表里是否被映射到同一个 Java 概念

### 1.2 【严重】00 1 天路径违反"坑点先行"原则

- **表现**:00 第 15 行明确说"**先看坑,再写代码**",但 1 天路径(`00 → 03 → 05 → 速查表`)完全跳过 06(坑点)
- **根因**:写 1 天路径时只考虑"时间紧 → 砍章节",没回头检查与开篇原则的冲突
- **修复**:1 天路径加显式提醒"**强烈建议花 15 分钟把 06 至少扫一遍深讲容器相关的 6 个坑**"
- **预防**:每个路径写完后,做一次"原则对账"——列 3-5 条开篇原则,逐条检查每条路径是否触发

### 1.3 【严重】00 1 天路径违反"ACM 模式 I/O 不再懵"目标

- **表现**:00 第 30 行明确写"不目标 ... 牛客 ACM 模式 I/O 不再懵",但 1 天路径完全跳过 01
- **根因**:同上,只看时间不看目标
- **修复**:加显式提醒"**牛客用户必看**:务必把 01 第七节 ACM 骨架先抄一遍"
- **预防**:见 1.2,做"目标对账"

### 1.4 【严重】01 标题"8 套模板"实际只有 7 套

- **表现**:`01_输入输出篇.md` 第 276 行写"8 套模板",但实际只有 7 套(模板 1 / 2a / 2b / 3 / 4 / 5 / 6)
- **根因**:subagent 自己说"2b 拆成两个变体凑成 8 套",但实际没拆,标题数错了
- **修复**:改"7 套"
- **预防**:**数字必须数清楚**——任何带数字的标题(`8 套` / `12 容器` / `15 大坑`),生成后必须 grep 实际数量对比

### 1.5 【严重】03 出现不存在的"LinkedSet"容器

- **表现**:`03_容器与STL对照.md` 第 1196 行写"**LinkedSet 必做**:LeetCode 146 LRU 缓存(用 LinkedList 维护访问顺序)"
- **根因**:subagent 误以为 Java 有 `LinkedSet`(实际只有 `LinkedHashSet` / `LinkedHashMap`),且推 LinkedList 做 LRU 是 O(n),而 `LinkedHashMap(accessOrder=true)` 是 O(1) 工业标准
- **修复**:改成"**LinkedHashMap 必做**:用 `new LinkedHashMap<>(capacity, 0.75f, true)` 三行实现",并在容器表新增 LinkedHashMap 行
- **预防**:**关键 API / 容器必须查官方文档确认存在**——不熟悉的 API 让 subagent 在生成前先 list 确认;LeetCode 高频题(前 100)的最优解法,要让 subagent 列出"该题的工业标准答案"再写

### 1.6 【严重】00 完全没铺垫"包装类/装箱"

- **表现**:00 总览表里直接出现 `ArrayList<T>` / `PriorityQueue<T>`,C++ 选手会问"T 是什么?为什么不是 int 而是 Integer?"
- **根因**:subagent 假设"装箱是常识",但对 C++ 选手来说**第一次见泛型就是要先铺垫**
- **修复**:在 00 对照表前加"2 分钟预备知识"小框(泛型必须对象 / `String` 不可变 / 方法签名 / `null` 数组)
- **预防**:**C++ → Java 文档必须显式铺垫装箱**——Java 泛型不能用基本类型,这是 C++ 模板(可以用 `int`)和 Java 泛型最大的语义差异,不放第一屏就会懵

### 1.7 【严重】01 模板 6 `buildTree` 函数过复杂

- **表现**:20 行代码一次混了"层序数组 → 二叉树 + 队列 BFS + 双指针下标"3 个新概念,C++ 选手第一次见会卡
- **根因**:subagent 把"完整可抄"和"易理解"对立,只追求前者
- **修复**:在 `null` 数组旁加注释,在 buildTree 上方加"思路"段(队列 BFS + 双指针 i 从 1 开始)
- **预防**:**关键代码块必须先讲思路再给代码**——尤其是混了多概念的(如 buildTree、BFS、DP 转移),不能只给代码

## 二、流程/方法论教训(6 条,直接影响后续阶段二)

### 2.1 必须派 subagent 审查,自己 review 不够

- **教训**:三篇生成后,我自己只做了基础行数/符号检查,没发现上述 7 个严重问题。**必须**派独立的 subagent 从"用户视角 + 事实核查"角度 review
- **做法**:每次让 subagent 生成 3 篇以上文档,生成后立即派 1 个 explore subagent 审查,输出结构化报告(评级 + 文件:行号 + 具体修正)
- **本次产出**:explore 报告 26 个问题,7 个严重全部命中

### 2.2 数字必须精确数清楚

- **教训**:subagent 会"凑数字"(`8 套模板`其实只有 7 套),用户第一眼会质疑文档专业度
- **做法**:生成后用 `rg -c '^### 模板' 文件.md` 或类似命令**精确计数**;任何带数字的标题都验证一遍
- **影响**:8 套 → 7 套,12 容器 → 13 容器(补 LinkedHashMap),10 大算法 → 10 大算法(已对齐),15 大坑 → 15 大坑(已对齐)

### 2.3 容器/API 必须查官方文档确认存在

- **教训**:"LinkedSet" 这种空气容器,subagent 编造时毫不犹豫——它不知道"不知道"
- **做法**:在 subagent prompt 里**显式要求**"任何不熟悉的容器/API 必须先 list 确认存在,不能凭印象写";对 Java 容器,要求 subagent 输出 `import java.util.*;` 后用到的类名清单,人工/工具查一遍
- **额外**:对 LeetCode 高频题(前 100)的"工业标准答案",subagent 必须明确"这道题最优解法是 X,用 Y 数据结构,原因 Z",不能拍脑袋

### 2.4 对照表必须做"内部一致性"自检

- **教训**:同一 C++ 概念在表里被映射到不同 Java 概念(`printf` → `String.format` vs `StringBuilder.append`),这种内部矛盾 subagent 不会自己发现
- **做法**:生成对照表后,选 5-10 个常见 C++ 概念(`printf` / `string` / `sort` / `pair` / `accumulate`),grep Java 列,看是否一致

### 2.5 路径设计必须做"原则对账"和"目标对账"

- **教训**:开篇原则(`坑点先行`)和文档目标(`ACM I/O 不再懵`)被 1 天路径无声违反
- **做法**:每个路径写完后:
  1. 列 3-5 条开篇原则 → 逐条检查是否触发
  2. 列 3-5 条速成目标 → 逐条检查是否满足
  3. 不满足的,在路径里加**显式提醒**(用 `> **⚠️ ...**` 块)

### 2.6 修复应聚焦高优先级,中低优先级先不做(避免范围蔓延)

- **教训**:explore 报告给了 26 个问题,7 高 / 13 中 / 6 低。一次修完会导致:1) 工作量爆炸 2) 高优先级修复被中低优先级噪音淹没 3) 用户看不清"核心修了什么"
- **做法**:**先修高优先级 7 个**(必须修,影响可用性),中低优先级**列清单**让用户决定要不要做,不要"顺便改"
- **本次实践**:7 个高全部修,中低 19 个列了清单等用户决策,没擅自改

- [ ] 所有提到的 Java 容器/API 确实存在于 `java.util` / `java.io` 等标准包
- [ ] LeetCode 高频题(前 50)的最优解法正确

- [ ] 关键代码块(多概念混合的)有思路段
- [ ] `null` / `Integer` / `String` 等 C++ 没有的细节有解释

### 跨章一致性
- [ ] 术语统一(优先队列/堆/PriorityQueue、二叉树/树/TreeNode、字符/字符串/char/String)
- [ ] 同一概念在三篇中说法一致(如装箱/拆箱、String 不可变)
- [ ] 路径与原则/目标对齐(坑点先行 / I/O 不再懵 / 核心章节)

### 风格与可执行性
- [ ] 错误/正确标记全文统一(用 `正确` 而非 `对`)
- [ ] 代码块语法正确(Java 高亮,小段 C++ 高亮)

- [ ] 章节末尾有"练习建议"块

## 五、根本性反思(给后续所有 subagent 任务的通用教训)

### 5.1 subagent 的"知识盲区"是隐形风险
- **症状**:subagent 会自信地写出"LinkedSet"这种不存在的容器,因为它不知道"不知道"
- **对策**:**关键事实(API 名 / 容器名 / 高频题最优解)必须让 subagent 显式 list 确认**,不能让它"凭印象写"

### 5.2 "完整可抄"和"易理解"经常冲突
- **症状**:subagent 追求"一个代码块覆盖所有场景",导致代码过长、概念过密
- **对策**:**关键代码块必须先讲思路再给代码**;多概念混合的(如 buildTree、BFS、DP)单独成节;长代码拆 3 个独立小例子

### 5.3 subagent 不会自己发现"内部矛盾"
- **症状**:对照表行间矛盾、路径与原则矛盾、标题与正文数字矛盾
- **对策**:**生成后必须派独立 subagent review**,专门查"跨行 / 跨节 / 跨章一致性",自己 review 抓不到

### 5.4 "用户视角"和"作者视角"差很多
- **症状**:subagent 假设读者知道"装箱",但 C++ 选手第一次见 `<Integer>` 必然懵
- **对策**:**铺垫要显式**——C++ → Java 文档必须显式铺垫:泛型必须对象 / `null` 数组 / 方法签名 / `String` 不可变

### 5.5 "风格确认问题"必须具体到可 yes/no
- **症状**:用户原本只回答了 1 个风格问题(对照表密度 OK),没回答另外 4 个
- **对策**:**风格确认 5 题应在 subagent 交付时一起给**,而不是等用户主动答;且问题必须可 yes/no,不能"够不够"这种开放题

## 六、给后续 subagent prompt 的强制条款(下次写 prompt 时加)

```markdown
## 生成前必须做的事
1. **API 存在性确认**:列出本文档将用到的所有 Java 容器/API,确保每个都在 `java.util` / `java.io` 等标准包存在
2. **高频题最优解确认**:LeetCode 前 50 高频题的最优解法和数据结构,必须显式 list(不熟悉的查官方题解)
3. **数字精确性**:任何带数字的标题(8 套 / 12 个 / 15 大),必须**生成后**用命令精确计数验证

## 生成后必须做的事
1. **对照表一致性自检**:选 5-10 个 C++ 概念,grep Java 列,看映射是否一致
2. **路径/原则对账**:每个路径写完后,列 3-5 条开篇原则和速成目标,逐条检查是否触发/满足
3. **用户视角可读性自检**:假设读者是"STL 半熟 + Java 零基础",看 5 个最容易卡住的地方,加铺垫

## 不允许做的事
- ❌ 写不存在的 Java 容器/API(如 LinkedSet)
- ❌ 写"凑数字"的标题(如 8 套但实际 7 套)
- ❌ 对照表出现同一 C++ 概念的不同 Java 映射
- ❌ 关键代码块只给代码不给思路
- ❌ 跳过"对 C++ 选手是新概念"的铺垫
```
