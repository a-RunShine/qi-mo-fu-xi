## 二、阶段二新发现的 11 个问题

### 2.1 【严重 / 高】04 文档 3 处方法嵌套在 main 里(Java 语法错误)

- **表现**:`04_常用工具类.md` 第 595 / 839 / 961 行,subagent 写了 3 个方法 `int reverse(int x) {}` / `public int firstUniqChar(String s) {}` / `public String tree2str(TreeNode root) {}`,**全部嵌套在 `public static void main` 内**
- **后果**:Java 不允许方法嵌套定义,用户直接复制会**编译失败**
- **根因**:subagent 想"演示 LeetCode 完整版",但用错了 Java 语法结构(Java 不像 JavaScript / Python,方法必须定义在类里)
- **修复**:
  - 3 个方法都抽出到 `class Solution { ... }` 块(在 main 代码块外)
  - main 内改为调用 `new Solution().reverse(x)` / `new Solution().firstUniqChar(s)` / 注释提示
- **预防**:**强制规则**:"完整可提交版必须在 `class Solution` 内,不能放在 main 里演示"

### 2.2 【严重 / 高】00 文档"7-8 个"和"12 个容器"过时

- **表现**:`00_总览与速成路线.md` 第 145 行写"02 必看的 7-8 个纯语法点",第 146 行写"03 的 12 个容器对照",但 02 实际有 10 个语法点,03 实际有 13 个容器
- **根因**:阶段一修了 02/03 的内容,但**00 表格里描述 02/03 的"一句话摘要"没同步更新**
- **修复**:00 第 145-146 行改为"10 个纯语法点"和"13 个容器对照"
- **预防**:**跨章数字改了要全文 grep 同步**——把"改 A 章节时检查所有引用 A 的章节"作为强制步骤

### 2.3 【严重 / 高】06 序言"C++ 选手必看 5 个"与实际标记数不符

- **表现**:06 第 5 行序言说"标【C++ 选手必看】的 5 个坑",但实际标了 3 个【C++ 选手必看】(坑 3/13/14)+ 2 个【C++ 选手第 X 坑】(坑 11/12)= 5 个但**两种标记方式**
- **根因**:坑 11/12 的标题用了"【C++ 选手第 1 坑】"格式(与"必看"区分),序言只数了"必看"标签
- **修复**:序言改"标【C++ 选手必看】或【C++ 选手第 X 坑】的 5 个坑(坑 3/11/12/13/14)"
- **预防**:**标记方式要统一**——要么全用"必看",要么全用"第 X 坑",不要混用

### 2.4 【中】03 LRU 简化版未标明"非最优"

- **表现**:`03_容器与STL对照.md` 第 895 行 LinkedList LRU 简化版,没标"非最优",而 03 第 37 行表格已声明 LinkedHashMap 是"LRU 缓存标准答案",文档内部自相矛盾
- **修复**:LinkedList LRU 代码上方加 `// ★ LRU 缓存(简化):用 LinkedList 维护访问顺序 —— O(n) **非最优** 工业标准:LinkedHashMap(accessOrder=true),见对照表第 37 行 + 速查表 13 容器`
- **预防**:**同一文档内不同方法对比,要明确标注优劣**

### 2.5 【中】04 第 6.5 节 multiply 缺 `class Solution` 包装

- **表现**:第 532-555 行 multiply 方法(LeetCode 43 字符串相乘)也是裸方法,Java 同样编译失败
- **修复**:**已在修复 2.1 时一起处理**(类似问题,统一规范)
- **预防**:**所有"完整可提交版"必须在 `class Solution` 内**

### 2.6 【中】风格标记不统一(02/03/04 用 `// 错误:`,06 用 `**错误**`)

- **表现**:02/03/04 代码注释用 `// 错误:` `// 正确:`(带冒号),06 段落标题用 `**错误**` `**正确**`(不带冒号)
- **修复**:跳过(中优先级,不影响可用性)
- **预防**:**风格指南要写明"错误/正确"标记的精确格式**,不是"用 `错误` / `正确` 标记"这种模糊措辞

### 2.7 【中】04 包装类缺"为什么需要"独立铺垫

- **表现**:04 第 7 节直接进 API 速查,虽然 03 已铺垫,但 04 自己作为独立章节缺一句"为什么需要包装类?"
- **修复**:跳过(中优先级,03 已铺垫够)
- **预防**:**章节级铺垫 vs 文档级铺垫**——04 作为独立文档,自己也要有 1-2 句总览

### 2.8 【低】03 第 465 行 `// 错` 简写不一致

- **表现**:同文档其他地方用 `// 错误:`,这里用 `// 错:`
- **修复**:跳过(低优先级,简写)
- **预防**:**同文档内风格严格统一**

### 2.9 【低】速查表 3.7 单调栈缺 LeetCode 编号

- **表现**:速查表第 3.7 节"下一个更大"是 LeetCode 496 风格,05 第 4.7.1 用的是 LeetCode 739(求距离)风格,两处不同
- **修复**:速查表标题加"LeetCode 496/739 风格"
- **预防**:**同一概念在不同章节,要明确标注 LeetCode 题号**

### 2.10 【低】速查表第 4 节关键字速查含 this / super

- **表现**:刷题几乎用不到 this / super
- **修复**:跳过(低优先级)
- **预防**:**速查表只列刷题用得到的**,不列完整关键字表

### 2.11 【低】风格标记统一(02/03/04 去掉冒号)

- **表现**:见 2.6
- **修复**:跳过
- **预防**:同 2.6

## 三、新发现的 2 个未在阶段一覆盖的问题(给后续 subagent 通用教训)

### 3.1 Java 语法规则必须显式约束

- **问题**:**Java 不允许方法嵌套定义在 main 里**(与 Python / JavaScript 不同),但 subagent 不知道,因为它可能用过 Python 写 demo
- **教训**:**跨语言迁移文档必须显式列出"目标语言的硬性语法规则"**,不能假设 subagent 知道
- **强制规则**(给后续 subagent prompt):
  ```
  Java 硬性语法规则(违反必编译失败):
  1. 方法不能嵌套定义(必须在类内,不能在其他方法内)
  2. 局部变量不能与成员变量同名
  3. final 变量必须初始化
  4. interface 方法不能 private
  ```

### 3.2 跨章数字漂移("改 A 时 B 没同步")

- **问题**:阶段一修了 02/03 的"几个语法点"/"几个容器"数字,但 **00 表格里描述 02/03 的一句话没同步**
- **教训**:**文档是网状引用,改一处要全文 grep 同步**
- **强制规则**(给后续 subagent prompt):
  ```
  改任何数字时,必须:
  1. grep 该数字的所有出现位置
  2. 同步更新所有引用
  3. 重点关注:摘要表、章节速查表、文档头部说明
  ```

## 七、给阶段三(未来)和其他 subagent 任务的直接建议

### 7.1 subagent prompt 必须包含 4 类规则
1. **目标语言硬性语法规则**(防 Java 语法错误)
2. **跨章引用规则**(防数字漂移)
3. **标记统一规则**(防风格混乱)
4. **API 存在性 + LeetCode 真实性**(防空气容器/假题号)

### 7.2 subagent review 必须包含 4 类检查
1. **数字精确性**:用 `rg` 精确计数
2. **跨章一致性**:grep 同一概念的所有出现
3. **API 存在性**:抽查 5-10 个 API
4. **代码可编译性**:每个代码块按 Java 语法规则验证

### 7.3 修复必须分级
- **高**:影响可用性(语法错 / 数字错 / 跨章矛盾)——必修
- **中**:影响质量(标记混乱 / 铺垫不足)——选择性修
- **低**:锦上添花(风格微调 / 细节优化)——跳过

### 7.4 时间分配
- 教训文档:1-2 分钟
- subagent 生成:5-10 分钟(并行)
- subagent 审查:3-5 分钟
- 修复高优先级:3-5 分钟
- 教训文档(本文件):3-5 分钟
- **总:15-25 分钟/批次**
