# 第五章 数据库完整性 — 选择题解析

---

### 第 1 题

**答案：B**

**解析：**

外码违约策略是 `ON DELETE CASCADE`（级联删除）。删 Student 表中 CS 学院的学生时，SC 表中引用这些学生的选课记录会被**级联删除**。

A 错：只说删了 Student，漏了 SC 的级联删除。
B 正确：删 Student + 级联删 SC 中对应选课记录。
C 错：CASCADE 策略不会拒绝，而是跟着删。
D 错：置空是 `SET NULL` 策略，不是 CASCADE。

> CASCADE = 父删子也删；SET NULL = 父删子置空；RESTRICT = 有引用就拒绝。三种策略别搞混。

### 第 2 题

**答案：C**

**解析：**

实体完整性规则：**主码唯一且不能取空值**。

A 错：主码**不能为空**。
B 错：复合主码不是"整体不为空就行"，而是**每个属性都不能为空**。
C 正确：复合主码的每个属性都不能为空——这是实体完整性的要求。
D 错：UNIQUE 约束**允许空值**，不满足实体完整性（实体完整性要求主码非空）。

> 主码 = 唯一 + 非空，两个条件缺一不可。UNIQUE 只保证唯一，允许空，所以不能替代主码。

---

### 第 3 题

**答案：C**

**解析：**

A 正确：外码可以为 NULL，表示"未引用"。
B 正确：外码可以引用被参照表中**有 UNIQUE 约束的列**，不一定是主码。
C **错误**：外码**不必**与被参照表的主码同名，只要语义对应即可（比如 SC.Sno 参照 Student.Sno 同名是巧合，也可以 Student.Sno 参照 People.ID 不同名）。
D 正确：外码非空时必须在被参照表中存在——这是参照完整性的核心。

> 外码的三自由：可以为空、可以引用 UNIQUE 列、可以不同名。唯一必须的是：非空时得在被参照表里找得到。

### 第 4 题

**答案：B**

**解析：**

CHECK 约束：`Grade >= 0 AND Grade <= 100`。

A：`Grade = NULL`——CHECK 约束遇到 NULL 时**视为满足**（UNKNOWN 当 TRUE），不会被拒绝。
B：`Grade = -1`——`-1 >= 0` 为假，违反 CHECK，**被拒绝**。选 B。
C：`Grade = 0`——`0 >= 0` 为真，满足。
D：`Grade = 100`——`100 <= 100` 为真，满足。

> CHECK 对 NULL 的处理是个坑：CHECK 只拒绝"明确为假"的，NULL 会被当成 UNKNOWN 放行。所以插 NULL 不会触发 CHECK 报错。

### 第 5 题

**答案：A**

**解析：**

A 正确：删除被参照表元组时，"拒绝"策略（RESTRICT/NO ACTION）——只要子表有引用就**不允许删除**。
B 错：级联删除是**删子表记录**，不是置空。置空是 SET NULL。
C 错：置空策略只置空**引用该父元组的子记录**的外码，不是所有子记录的外码都置空。
D 错：三种策略（拒绝/级联/置空）是**三选一**，不能同时使用。

> 拒绝 = 不让删；级联 = 跟着删；置空 = 子记录外码设 NULL。针对同一个外码只能选一种。

### 第 6 题

**答案：C**

**解析：**

关键区分：**属性级 CHECK** 只能检查单个属性的值，**元组级 CHECK** 可以检查同一行内多个属性之间的关系。

A：年龄 15-60——只涉及 Sage 一个属性，属性级 CHECK 够用。
B：学分 > 0——只涉及 Ccredit 一个属性，属性级够用。
C：发货日期不能早于下单日期——涉及**两个属性**（发货日期、下单日期）的关系比较，**必须用元组级 CHECK**。选 C。
D：手机号 11 位——只涉及 phone 一个属性，属性级够用。

> 涉及"一个属性的值范围"用属性级；涉及"同一行两个属性的关系"必须用元组级。

### 第 7 题

**答案：C**

**解析：**

A 正确：一个表只能一个主码，但可有多个 UNIQUE。
B 正确：主码非空，UNIQUE 允许空。
C **错误**：PRIMARY KEY 和 UNIQUE **都会自动创建索引**（主码自动建主索引，UNIQUE 自动建唯一索引）。说"UNIQUE 不会"是错的。
D 正确：主码标识实体，UNIQUE 保证不重复，定位不同。

> 主码和 UNIQUE 都建索引，这是共同点。区别在：主码只能一个且非空，UNIQUE 可多个且允许空。

### 第 8 题

**答案：B**

**解析：**

Orders 表的 CustomerID 是外码（参照 Customers.CustomerID）。出现 CustomerID='C999' 的订单但 Customers 表无此客户——**外码值在父表找不到对应主码**，违反**参照完整性**。

A 错：实体完整性是主码问题，这里主码 OrderID 没问题。
B 正确：外码找不到对应主码 = 参照完整性被破坏。选 B。
C 错：用户定义完整性是 CHECK 之类的业务规则，这里没涉及。
D 错：这是完整性问题，不是安全性问题。

> 父表找不到对应主码 = 参照完整性。就像你填的紧急联系人电话在通讯录里查无此人。

### 第 9 题

**答案：C**

**解析：**

A 错：不是都在提交时检查，大部分在**语句执行时**就检查。
B 错：不是按完整性类型分时机，而是基本都在语句执行时检查。
C 正确：完整性约束通常在 **INSERT/UPDATE/DELETE 语句执行时**检查——这是标准做法。
D 错：不是启动时检查一次，而是每次写操作都检查。

> 完整性检查是"实时"的——每条 INSERT/UPDATE/DELETE 执行时就查，不用等到提交。

### 第 10 题

**答案：B**

**解析：**

违约策略 `ON DELETE SET NULL`：删除 Dept 中 Dno='D01' 的部门时，Employee 表中 Dno='D01' 的员工的 Dno 被**置空**（设为 NULL）。

A 错：还是 'D01' 没变——那是没做任何处理。
B 正确：置空为 NULL。选 B。
C 错：被删除——那是 CASCADE 的行为。
D 错：报错拒绝——那是 RESTRICT 的行为。

> SET NULL = 父删子置空。员工还在，只是部门字段变 NULL（变成"无部门"员工）。
