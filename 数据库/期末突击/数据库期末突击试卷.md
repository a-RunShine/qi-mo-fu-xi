## 六、关系数据理论题（12 分，第 6 章）

**21.** 已知关系模式 R(U, F)，其中：

- U = { S（学号）, C（课程号）, G（成绩）, T（教师号）, TN（教师姓名）, CN（课程名）, CR（学分）, D（开课院系） }
- F = {
    (S, C) → G,
    (S, C) → T,
    T → TN,
    T → D,
    C → CN,
    C → CR,
    C → D,
    CN → CR
  }

**请围绕上述 U 与 F 回答下列 3 小问**：

**（1）[候选码 + 闭包，4 分]**

- a) 用 L/R/LR 分类法指出 S、C、G、T、TN、CN、CR、D 中各自属于哪一类；（1 分）
- b) 求 R 的**所有候选码**；（1 分）
- c) 求属性组 (S, C) 在 F 下的**闭包**，并写出每一步推导过程。（2 分）

**（2）[范式判定，4 分]** 依次判断 R 是否满足 1NF / 2NF / 3NF / BCNF，对每一级范式给出"满足 / 不满足"的结论与简要理由（若不满足，请指出违反了哪一条函数依赖）。写出 R 的**最高范式级别**。

**（3）[模式分解，4 分]** 将 R 分解为**满足 3NF 且保持函数依赖**的一组关系模式。

**要求**：

- a) 先求出 F 的**最小函数依赖集 Fmin**（写出拆分右侧、去除自身、左侧最小化三步的中间结果）；（1 分）
- b) 按"分组 + 候选码"算法给出 3NF 分解；（2 分）
- c) 说明所得到的分解**是否同时具有无损连接性**（理由可基于"分解中包含候选码"这一充分条件）。（1 分）

## 七、数据库设计题（10 分，第 7 章+5 章）

**22.（E-R + 关系模式 + 完整性约束 + 触发器）** —— 业务场景：**校园外卖配送系统**

某高校拟建设校园外卖配送系统，需要管理以下信息：

- **用户**（学生/教职工）：用户号、姓名、手机号、所在宿舍楼（或办公室地址）
- **商家**（食堂档口/校内外店铺）：商家号、商家名、商家地址、营业状态（营业中/休息中）
- **菜品**：菜品号、菜品名、单价、所属商家
- **骑手**（仅限在校学生兼职）：骑手号、姓名、手机号、当前状态（空闲/配送中/休息）
- **订单**：订单号、所属用户、所属商家、下单时间、期望送达时间、配送地址、订单状态（待接单/已接单/配送中/已完成/已取消）、订单总金额
- **配送**（订单与骑手的多对多联系）：一份订单可被多位骑手尝试接单，但**最终仅一位**骑手成功配送；一次配送记录包含接单时间与送达时间

**业务约束**：

- 每个菜品必须属于**恰好一个**商家；
- 每张订单必须属于**恰好一个**用户与**恰好一个**商家；
- 订单总金额 > 0；期望送达时间 ≥ 下单时间；骑手手机号必须为 11 位数字。

请完成以下 4 小问：

**（1）[画 E-R 图，4 分]** 画出该系统的 E-R 图。要求：

- 用矩形表示实体、椭圆表示属性、菱形表示联系；
- 标出实体之间的联系类型（1:1 / 1:N / M:N）；
- 关键属性请在实体旁注明下划线（主码）。

**（2）[E-R 转关系模式，3 分]** 将上述 E-R 图转换为关系模式集合。要求：

- 每个实体转换为一个关系模式；
- m:n 联系"配送"独立转换为一个关系模式；
- 标出每个关系模式的**主码**与**外码**（外码需注明引用哪个关系模式）。

**（3）[完整性约束 SQL，2 分]** 用 `CREATE TABLE` 写出"订单"关系模式的 DDL 语句，要求体现：

- **实体完整性**：订单号为主码；
- **参照完整性**：所属用户、所属商家为外码，分别引用用户表与商家表，删除时采用 **RESTRICT** 策略；
- **用户定义完整性**：订单总金额 CHECK (> 0)、期望送达时间 ≥ 下单时间（可使用 CHECK 约束或触发器，**任选其一**即可）。

**（4）[触发器，1 分]** 当订单状态从'配送中'变为'已完成'时，要求自动将对应骑手的"当前状态"置为'空闲'。请写出**触发器**的 `CREATE TRIGGER` 语句主体（不要求写出 DELIMITER，使用标准 SQL/PSM 语法即可）。
