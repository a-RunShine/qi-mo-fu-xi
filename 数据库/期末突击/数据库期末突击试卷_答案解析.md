## 第 16 题（第 1 章，5 分）

### 答案要点

**三级模式**：

1. **外模式（子模式 / 用户模式）**：单个用户或应用所能看到的**那部分局部数据**的逻辑结构，面向用户/应用程序。
2. **模式（概念模式 / 逻辑模式）**：数据库**全体数据**的全局逻辑结构与特征描述，面向全体用户。
3. **内模式（存储模式）**：数据在磁盘上的**物理存储结构与存取方式**（如 B+ 树、堆文件、索引组织），面向 DBMS 内部。

**二级映像与独立性**：

4. **外模式/模式映像**：定义外模式与模式之间的对应关系。**当模式改变时**（如表拆分、增加字段），通过修改映像即可使外模式保持不变 → **保证数据的逻辑独立性**。
5. **模式/内模式映像**：定义模式与内模式之间的对应关系。**当内模式改变时**（如换索引、改存储结构），通过修改映像即可使模式保持不变 → **保证数据的物理独立性**。

### 详细解析

- 三级模式 + 二级映像是数据库系统实现"数据独立性"的核心机制。**逻辑独立性**意味着"全局结构变了，应用程序不用改"；**物理独立性**意味着"存储方式变了，全局结构/应用程序都不用改"。
- 易混点：模式（schema）一词在不同语境含义不同——本章的"模式"是"全局逻辑结构"，不是"模式（SCHEMA）对象"。考试时若题目问"CREATE SCHEMA 的 SCHEMA 是什么"，答案是"命名空间"，与本处的"模式"概念不同。

### 评分标准（5 分）

| 给分点 | 分值 | 评分细则 |
|--------|:----:|----------|
| 外模式含义 | 1 | 写出"局部/用户级逻辑结构"，少 1 关键词扣 0.5 |
| 模式含义 | 1 | 写出"全局/整体逻辑结构" |
| 内模式含义 | 1 | 写出"物理/存储结构" |
| 逻辑独立性对应 | 1 | "外模式/模式映像"+"逻辑独立性"，二者都写出才给分 |
| 物理独立性对应 | 1 | "模式/内模式映像"+"物理独立性"，二者都写出才给分 |

> **"内对外是物理，外对内是逻辑"** —— 反过来记：内模式变 → 上面的程序不变，是物理独立性；模式（全局结构）变 → 外面的程序不变，是逻辑独立性。

## 第 17 题（第 4 章，5 分）

### 答案要点

**（1）执行完上述语句后的权限（1 分）**：

- **李明**：对 Course 表拥有 `SELECT` 和 `UPDATE` 权限，且这两项权限**可再转授他人**（因带 WITH GRANT OPTION）。
- **王芳**：对 Course 表拥有 `SELECT` 权限（从李明处获得），**不可再转授**（因李明在转授时未带 WITH GRANT OPTION）。

**（2）执行 `REVOKE ... CASCADE` 后的变化（2 分）**：

- **李明的 SELECT 权限被回收**（教务主任显式回收李明的 SELECT）；李明仍保留 UPDATE 权限（本次未回收 UPDATE）。
- **王芳的 SELECT 权限被级联回收**（她是从李明处获得的 SELECT，李明失去了 SELECT 授权能力，她凭此获得的 SELECT 自动失效）。
- **CASCADE 的语义**：级联回收由该用户传播出去的所有授权（即"砍树枝"——不仅砍被回收者本身，还砍从他这里长出去的枝叶）。

**（3）改为 RESTRICT 的行为（2 分）**：

- DBMS 检查到李明**已经将 SELECT 转授给了王芳**，因此**拒绝执行**整个 REVOKE 语句（不进行任何回收，李明的 SELECT 仍保留）。
- **前提条件**：要使 RESTRICT 成功执行，必须**先撤销**李明对王芳的转授（让李明不再"传播"过 SELECT 权限），或者**先回收**王芳的 SELECT 权限。

### 详细解析

- 关键概念辨析：
  - **CASCADE** = 砍树枝。被回收者失去权限，所有从他这里获得该权限的下游用户也连带失去（是"级联回收"）。
  - **RESTRICT** = 金钟罩 + 紧箍咒。发现下游有传播就拒绝执行（"宁死不屈"）。
  - 两种模式在 DB2 / Oracle / PostgreSQL / SQL Server 中都有支持；MySQL 不支持 RESTRICT 模式。
- 本题中 UPDATE 权限不受影响的关键：教务主任**只回收了 SELECT**，没回收 UPDATE；李明仍可凭 UPDATE 权限的 WITH GRANT OPTION 后续操作。

### 评分标准（5 分）

| 给分点 | 分值 | 评分细则 |
|--------|:----:|----------|
| (1) 李明与王芳权限 | 1 | 李明：SELECT+UPDATE+可转授；王芳：SELECT+不可转授。漏"可/不可转授"扣 0.5 |
| (2) CASCADE 后变化 | 1 | 李明 SELECT 失去、王芳 SELECT 失去、UPDATE 保留 |
| (2) CASCADE 语义解释 | 1 | 写出"级联回收"或"砍树枝"等关键表述 |
| (3) RESTRICT 行为 | 1 | "拒绝执行"或"不进行任何回收" |
| (3) RESTRICT 成功前提 | 1 | "无传播"或"需先回收王芳的 SELECT" |

### 金句

> **"CASCADE 砍树枝，RESTRICT 金钟罩"** —— 砍树枝：连根带叶一起收；金钟罩：发现下游有传播就硬扛拒绝。

## 第 18 题（第 5 章，5 分）

### 答案要点

**4 种破坏情形**：

1. **向 SC 插入元组**：外码 `SC.Sno` 的值在 Student 表中**不存在**（孤儿引用）。
2. **删除 Student 元组**：SC 表中仍有引用该 `Student.Sno` 的元组（被引用行消失）。
3. **修改 Student.Sno**：SC 表中引用该 `Student.Sno` 的外码值变为"孤儿"（被引用主码值变了）。
4. **修改 SC.Sno**：新 Sno 值在 Student 表中**不存在**（修改后的外码变孤儿）。

**3 种违约处理策略及适用**：

- **拒绝执行（NO ACTION / RESTRICT）**：上述 4 种情形**均可**使用，是**子表（SC）插入/更新违约**的唯一可选策略（因子表动作不影响父表），也是最保守的默认。
- **级联操作（CASCADE）**：适用于情形 2、3 —— 父表删除/更新被参照列时，**子表对应元组一并删除/级联更新**（"父动子随动"）。
- **置空值（SET NULL）**：适用于情形 2、3 —— 父表删除/更新被参照列时，**子表对应外码置为 NULL**（前提是该外码允许为空，否则拒绝）。

### 详细解析

- 4 种情形的本质是"**子表外码值在父表主码中找不到对应项**"。从子表/父表两个角度 × 增/删/改两个动作 = 4 组合。
- 关键规律：**情形 1、4 是子表动作**（SC 自己改）→ 违约处理只能"拒绝"（你不能在子表插入时去级联删父表，那样更乱）；**情形 2、3 是父表动作**（Student 改）→ 违约处理三选一（拒绝、级联、置空）。
- SET NULL 的局限性：要求外码列**允许 NULL**，否则无法置空，会自动回退为"拒绝"。

### 评分标准（5 分）

| 给分点 | 分值 | 评分细则 |
|--------|:----:|----------|
| 4 种情形完整 | 2 | 每种情形 0.5 分；缺一扣 0.5；表述不全（如只说"插入"不说"外码值不存在"）扣 0.25 |
| 拒绝执行适用 | 1 | 写明"4 种情形均适用"或"子表违规唯一策略" |
| 级联操作适用 | 1 | 写明"情形 2、3"+"父表删除/更新时子表随动" |
| 置空值适用 | 1 | 写明"情形 2、3"+"SET NULL"+"前提是外码允许空" |

### 金句

> **"子表违规拒绝，父表违规三选"** —— 子表主动改自己 → 只能拒绝；父表主动改自己 → 拒绝/级联/置空三选一。
>
> **"父动子随动"** —— 父表删除/更新时，级联策略会让子表对应行也删除/更新；置空策略会让子表对应外码变 NULL。

## 第 19 题（关系代数综合，8 分，第 2 章）

### 答案要点

**（1）[基础，2 分]** 查询"上海"分行的分行号与分行名：

```
π 分行号,分行名 (σ 城市='上海' (B))
```

- σ 负责**过滤行**（按"城市='上海'"筛选 B 表）；
- π 负责**投影列**（从筛选结果取"分行号"与"分行名"两列）。

**（2）[中等，3 分]** 查询在上海分行开户且余额>5万的客户姓名与余额：

```
π 姓名,余额 (σ 城市='上海' ∧ 余额>50000 (B JOIN A JOIN C))
```

- 三表自然连接：B JOIN A（按分行号 = 开户分行号）、A JOIN C（按客户号 = 客户号）；
- σ 一次性过滤两条件（连接后同时按"城市='上海'"和"余额>50000"筛选）；
- π 投影姓名与余额两列。
- **排序说明**：关系代数层**不保证**输出顺序，**排序属于结果呈现层**；翻译为 SQL 时应在末尾加 `ORDER BY 余额 DESC`。

**（3）[高，3 分]** 查询"在所有上海分行都开过户"的客户姓名 —— **典型"全部"语义，使用除运算 ÷**：

```
π 姓名 (C JOIN (π 客户号,开户分行号 (A) ÷ π 分行号 (σ 城市='上海' (B))))
```

- **被除关系** = `π 客户号,开户分行号 (A)` —— 每位客户在每个分行开户的明细（每位客户的所有开户分布）；
- **除关系** = `π 分行号 (σ 城市='上海' (B))` —— 所有上海分行的编号集合；
- 除运算 `÷` 的语义：A 中那些**与除关系中每个分行都配对**的客户号，即"在所有上海分行都开过户"的客户；
- 最后 JOIN C 取姓名，π 投影出最终客户姓名。

### 详细解析

- **除运算的判别关键**：题干含"**所有 / 全部 / 都**"时使用除运算；题干含"**至少一个 / 任一**"时不能用除。本题"在**所有**上海分行都开过户"是除运算的典型场景。
- **陷阱**：易错写为"在某个上海分行开过户"（用 σ 即可），或写为 JOIN+GROUP BY+HAVING（SQL 等价改写可行但关系代数层用 ÷ 更直接）。
- **(1) 与 (2) 的本质区别**：(1) 是单表 σ+π；(2) 涉及多表连接 + 多条件 σ + 投影；(3) 涉及除运算（最高难度）。三小问覆盖 σ、π、JOIN、÷ 四种基本运算中的三种（差、并、交、笛卡尔积、聚合未直接考察）。

### 评分标准（8 分）

| 给分点 | 分值 | 评分细则 |
|--------|:----:|----------|
| (1) σ 正确 | 1 | 条件 `城市='上海'` 写对 |
| (1) π 正确 | 1 | 投影列写对 |
| (2) 三表 JOIN 正确 | 1 | BJOINAJOINC 或等价连接顺序，连接条件隐含于自然连接 |
| (2) σ 多条件正确 | 1 | "城市='上海' ∧ 余额>50000" 两条件合取 |
| (2) π 正确 | 0.5 | 投影"姓名,余额"两列 |
| (2) 排序说明 | 0.5 | 明确说明"排序属于 SQL 层 / 关系代数不保证顺序" |
| (3) 被除关系 | 1 | `π 客户号,开户分行号 (A)` |
| (3) 除关系 | 1 | `π 分行号 (σ 城市='上海' (B))` |
| (3) ÷ 运算符与 JOIN C 配合 | 1 | 完整表达式结构正确，姓名取自 C |

### 金句

> **"全部用除，否定用差"** —— "所有/全部/都" → ÷；"没有/不/未" → −。
>
> **"除运算 = 找满足全部条件的人"** —— 找的是"满足除关系中所有项的元组"。

## 第 20 题（SQL 综合，10 分，第 3 章）

### 答案要点

**（1）[基础，3 分]** 上海金卡/钻石等级顾客 2024 年订单 Top10：

```sql
SELECT H.订单号, K.姓名, H.总金额
FROM K JOIN H ON K.顾客号 = H.顾客号
WHERE K.城市 = '上海'
  AND K.等级 IN ('金卡', '钻石')
  AND H.下单时间 BETWEEN '2024-01-01' AND '2024-12-31'
ORDER BY H.总金额 DESC
LIMIT 10;
```

- JOIN：顾客与订单按顾客号连接；
- WHERE：城市 + 等级（IN 包含多值）+ 下单时间范围（2024 全年）；
- ORDER BY ... DESC：按总金额降序；
- LIMIT 10：取前 10 条。

**（2）[中等，4 分]** 每类商品累计销售数+金额（已付款/已发货/已完成）：

```sql
SELECT P.类别,
       SUM(D.数量)        AS 累计销售数量,
       SUM(D.小计金额)    AS 累计销售金额
FROM P
JOIN D ON P.商品号 = D.商品号
JOIN H ON D.订单号 = H.订单号
WHERE H.状态 IN ('已付款', '已发货', '已完成')
GROUP BY P.类别
HAVING SUM(D.数量) >= 100
ORDER BY 累计销售金额 DESC;
```

- 多表连接顺序：P JOIN D JOIN H（商品→订单明细→订单）；
- WHERE 过滤状态（IN 包含 3 个值）；
- GROUP BY 按类别分组；
- 聚合函数 SUM 同时计算数量与小计金额；
- HAVING 过滤分组（累计数量 ≥ 100）；
- ORDER BY 按累计销售金额降序。

**（3）[高/新造，3 分]** 存储过程 `proc_category_sales`：

```sql
CREATE PROCEDURE proc_category_sales(
    IN  p_category     VARCHAR(20),
    OUT p_total_qty    INT,
    OUT p_total_amount DECIMAL(12,2)
)
BEGIN
    DECLARE v_count INT DEFAULT 0;

    -- 1) 统计符合条件的明细条数（用于 IF 兜底判断）
    SELECT COUNT(*)
      INTO v_count
      FROM D
      JOIN P ON D.商品号 = P.商品号
      JOIN H ON D.订单号 = H.订单号
     WHERE P.类别 = p_category
       AND H.状态 IN ('已付款', '已发货', '已完成');

    -- 2) IF 流程控制：无数据时置零，否则用 SELECT INTO 汇总
    IF v_count = 0 THEN
        SET p_total_qty = 0;
        SET p_total_amount = 0.00;
    ELSE
        SELECT COALESCE(SUM(D.数量), 0),
               COALESCE(SUM(D.小计金额), 0.00)
          INTO p_total_qty, p_total_amount
          FROM D
          JOIN P ON D.商品号 = P.商品号
          JOIN H ON D.订单号 = H.订单号
         WHERE P.类别 = p_category
           AND H.状态 IN ('已付款', '已发货', '已完成');
    END IF;
END;
```

4 要素逐项核验：
- **参数声明**：`IN p_category VARCHAR(20)` + `OUT p_total_qty INT` + `OUT p_total_amount DECIMAL(12,2)` 【对】
- **DECLARE 局部变量**：`DECLARE v_count INT DEFAULT 0;` 【对】
- **SELECT INTO 赋值**：两处（`SELECT COUNT(*) INTO v_count` 与 `SELECT SUM(...) INTO p_total_qty, p_total_amount`）【对】
- **IF 流程控制**：`IF v_count = 0 THEN ... ELSE ... END IF;` 【对】

### 详细解析

- **(1) 易错点**：
  - 漏 LIMIT 10：未限定前 10 条；
  - 下单时间范围错写：用 `YEAR(下单时间) = 2024` 也可，但 BETWEEN 写日期字面量更直观；
  - 等级 IN 漏值：'金卡','钻石' 写漏一个。

- **(2) 易错点**：
  - 漏 JOIN H：缺少"订单状态"过滤维度；
  - GROUP BY 写类别时漏 `P.类别`：导致分组错误；
  - HAVING 与 WHERE 混淆：聚合函数必须放 HAVING，不能放 WHERE；
  - 漏状态过滤：写成"所有订单"含'待付款'与'已取消'，业务不对。

- **(3) 关键语法点**：
  - `CREATE PROCEDURE` 是标准 SQL/PSM 语法（也兼容 MySQL）；
  - `IN` 参数只读传入，`OUT` 参数用于返回值；
  - `SELECT ... INTO` 用于把查询结果赋值给变量（替代 RETURN，多个变量用逗号分隔）；
  - `IF ... THEN ... ELSE ... END IF;` 是标准 PSM 流程控制；
  - `COALESCE(SUM(...), 0)` 用于 SUM 返回 NULL 时兜底为 0（防止无数据时返回 NULL）；
  - 局部变量声明必须在 BEGIN 后、C 引用前；`DEFAULT 0` 是初值。

### 评分标准（10 分）

| 给分点 | 分值 | 评分细则 |
|--------|:----:|----------|
| (1) JOIN + WHERE | 1 | K JOIN H + 三条件（城市/等级/时间）齐全 |
| (1) ORDER BY + LIMIT | 1 | 排序字段 + LIMIT 10 |
| (1) 字段投影正确 | 1 | "订单号, 姓名, 总金额" 三列 |

| (2) HAVING + ORDER BY | 1 | `HAVING SUM(D.数量) >= 100` + `ORDER BY 累计销售金额 DESC` |

> **"存储过程 = 函数 + SQL"** —— 参数 (IN/OUT) + 局部变量 (DECLARE) + SQL 语句 (SELECT INTO) + 流程控制 (IF/WHILE) + 异常处理 (HANDLER)。

## 第 21 题（关系数据理论，12 分，第 6 章）

### 答案要点

**（1）[候选码 + 闭包，4 分]**

**a) L/R/LR 属性分类（1 分）**：

| 属性 | 在 F 中出现位置 | 类别 |
|------|------------------|------|
| S    | 仅左侧 `(S,C)→G, (S,C)→T` | **L** |
| C    | 仅左侧 `(S,C)→G, (S,C)→T, C→CN, C→CR, C→D` | **L** |
| G    | 仅右侧 `(S,C)→G` | **R** |
| T    | 左右均有 `(S,C)→T, T→TN, T→D` | **LR** |
| TN   | 仅右侧 `T→TN` | **R** |
| CN   | 左右均有 `C→CN, CN→CR` | **LR** |
| CR   | 仅右侧 `C→CR, CN→CR` | **R** |
| D    | 仅右侧 `T→D, C→D` | **R** |

分类汇总：**L = {S, C}**，**R = {G, TN, CR, D}**，**LR = {T, CN}**。

**b) 候选码（1 分）**：

候选码必包含 L 类属性 `{S, C}`（L 类必在码中）。计算 (S, C) 的闭包看是否等于 U：

- 初始 X = {S, C}
- 由 `(S, C) → G`，加入 G → X = {S, C, G}
- 由 `(S, C) → T`，加入 T → X = {S, C, G, T}
- 由 `T → TN`，加入 TN → X = {S, C, G, T, TN}
- 由 `T → D`，加入 D → X = {S, C, G, T, TN, D}
- 由 `C → CN`，加入 CN → X = {S, C, G, T, TN, D, CN}
- 由 `C → CR`，加入 CR → X = {S, C, G, T, TN, D, CN, CR}
- 由 `C → D`（D 已含）
- 由 `CN → CR`（CR 已含）
- 终态 X = {S, C, G, T, TN, D, CN, CR} = U 【对】

**所以 R 的唯一候选码是 (S, C)**。

**c) (S, C) 的闭包（2 分）**：

(S, C)^+ = {S, C, G, T, TN, D, CN, CR} = U（每步推导同 b 中所列）。

**（2）[范式判定，4 分]**

- **1NF 满足**（默认所有属性不可再分）
- **2NF 不满足**：因为 C → CN、C → CR、C → D 中，C 是候选码 (S, C) 的**真子集**，所以非主属性 CN、CR、D 对码存在**部分函数依赖**（部分依赖码）—— 这违反 2NF 的定义。
- **3NF 不满足**：因为 T → TN、T → D 中，T 不是候选码，但 (S, C) → T → TN 构成**传递函数依赖**（(S, C) → T → TN、(S, C) → T → D）—— 违反 3NF。
- **BCNF 不满足**：因为 T → TN、T → D、C → CN、C → CR、C → D 中，**决定因素 T 和 C 都不含候选码**—— BCNF 要求"每个依赖的左侧都包含候选码"。

**最高范式级别：1NF**。

**（3）[模式分解为 3NF，4 分]**

**a) 最小函数依赖集 Fmin（1 分）**：

对 F 应用三步化简：

1. **Step 1 拆分右侧**：F 中各 FD 右侧已均为单属性，无需拆分。F1 = F。
2. **Step 2 去除自身**（对每个 FD `X→A`，求 X 在 F - {X→A} 下的闭包，若已含 A 则该 FD 冗余）：
   - `(S,C)→G`：去掉后 (S,C)^+ 推不出 G（虽然有 `(S,C)→T` 但推不到 G），**保留**。
   - `(S,C)→T`：去掉后 (S,C)^+ 推不出 T，**保留**。
   - `T→TN`：去掉后 T^+ 推不出 TN，**保留**。
   - `T→D`：去掉后 T^+ 推不出 D，**保留**。
   - `C→CN`：去掉后 C^+ 推不出 CN，**保留**。
   - `C→CR`：去掉后 C^+ 推不出 CR，**保留**。
   - `C→D`：去掉后 C^+ 推不出 D（C → CN → CR 但推不到 D），**保留**。
   - `CN→CR`：去掉后 CN^+ 推不出 CR（CN 不能再推任何东西），**保留**。
   - F2 = F
3. **Step 3 左侧最小化**（对每个 FD `X→A`，X 中是否有多余属性可去）：
   - `(S,C)→G`：S^+ = {S} 不含 G；C^+ 不含 G（虽然 C→CN/CR/D）；**两边都不可单独推 G**，**保留 (S,C)**。
   - `(S,C)→T`：同理两边都不可单独推 T，**保留 (S,C)**。
   - 其他 FD 左侧均单属性，无需左侧化简。
   - F3 = F

**Fmin = F**（与原 F 实质相同，仅经形式化处理）。

**b) 3NF 分解（2 分）**：

按"分组 + 候选码"算法——把 Fmin 中**决定因素相同的依赖划为一组**：

- `(S,C) → G`、`(S,C) → T` → 一组：**R1(S, C, G, T)**（候选码 = (S, C)）
- `T → TN`、`T → D` → 一组：**R2(T, TN, D)**（候选码 = T）
- `C → CN`、`C → CR`、`C → D` → 一组：**R3(C, CN, CR, D)**（候选码 = C）
- `CN → CR` → 独立：**R4(CN, CR)**（候选码 = CN；与 R3 共享 CN，但因 R3 决定因素是 C 而非 CN，保留 R4 更稳妥）

**候选码已在 R1 中**（R1 主码即 (S, C)），不必再单独加 R5。

**最终 3NF 分解**：`{R1(S, C, G, T), R2(T, TN, D), R3(C, CN, CR, D), R4(CN, CR)}`

**c) 无损连接性判定（1 分）**：

根据充分条件：**若分解中包含原关系模式的候选码，则该分解必具有无损连接性**。本分解 R1 的主码为 (S, C)，恰为 R 的候选码。

**所以该分解具有无损连接性**。

此外，该分解保留了 Fmin 中全部 FD（每条 FD 都能在某个分解关系中检查），**同时具有保持函数依赖性**。

### 详细解析

- **关键 F 设计意图**：本题 F 故意构造为"含部分依赖 + 含传递依赖 + 决定因素不含码"，最高范式仅 1NF，分解步骤完整覆盖 6 步上篮法的核心考点。
- **陷阱点**：
  - 候选码唯一性：学生容易误把 T 或 C 当候选码。判定方法——L 类属性必在码中，本题 L={S,C}，起点必含 S 和 C。
  - 2NF 违反：常被误判为满足。检查方法——找"决定因素是候选码真子集"且"右侧是非主属性"的 FD，本题 C→CN/CR/D 均符合。
  - 3NF 违反：常被漏判。检查方法——找"非主属性传递依赖候选码"，本题 (S,C)→T→TN 即传递。
  - BCNF 违反：易被遗漏。检查方法——B**所有** FD 左侧都要含候选码，**含主属性的依赖也要查**（本题 T→TN 中 T 不是候选码）。
- **Fmin 化简陷阱**：本题 F 已是最小形式（无冗余依赖、无左侧多余属性）。学生若答"化简后无变化"是正确答案。
- **3NF 分解 vs BCNF 分解**：3NF 允许"决定因素不含码"（如 T→TN 中 T 不含码），只要求"无部分依赖、无传递依赖（针对非主属性）"；BCNF 要求"所有决定因素都含码"。本题因违反 BCNF，**3NF 分解后仍含 R2(T, TN, D)**（T 不含候选码）—— 这正是 3NF 而非 BCNF 的体现。

### 评分标准（12 分）

| 给分点 | 分值 | 评分细则 |
|--------|:----:|----------|
| (1)a L/R/LR 分类 | 1 | L={S,C}、R={G,TN,CR,D}、LR={T,CN}，三档各列对 |
| (1)b 候选码 | 1 | (S,C)，理由充分（指出 (S,C)+ = U） |
| (1)c 闭包推导 | 2 | 每步 FD 应用 + 终态=U，分步清晰 |
| (2) 1NF | 0.5 | 满足，理由：默认 |
| (2) 2NF | 1 | 不满足，理由：C→CN/CR/D 部分依赖（指明 C 是码的真子集） |
| (2) 3NF | 1 | 不满足，理由：(S,C)→T→TN/D 传递依赖 |
| (2) BCNF | 1 | 不满足，理由：T→TN/D、C→CN/CR/D 中决定因素不含码 |
| (2) 最高范式 | 0.5 | 1NF |
| (3)a Fmin | 1 | Fmin = F（三步过程写明，本题 F 已最小） |
| (3)b 3NF 分解 | 2 | R1/R2/R3/R4 完整，组内 FD 划分正确 |
| (3)c 无损连接判定 | 1 | 引用"含候选码⇒无损"充分条件 |

### 金句

> **"L 类必在码中，R 类绝不在码中，LR 类逐个试"** —— 属性分类法判定候选码起点的口诀。
>
> **"2NF 看非主，3NF 拦传递，BCNF 管得宽"** —— 2NF 只看非主属性是否完全依赖码；3NF 在 2NF 基础上拦非主属性的传递依赖；BCNF 管所有依赖（含主属性）。
>
> **"分解含候选码 ⇒ 必无损"** —— 3NF 分解时如果包含了原关系候选码所在的子关系，则该分解必有无损连接性。

## 第 22 题（数据库设计，10 分，第 7 章+5 章）

### 答案要点

**（1）[画 E-R 图，4 分]**

E-R 图参见 `数据库期末突击试卷_ER图参考.png`。参考图结构：

- **6 个实体**（矩形）：用户、商家、菜品、骑手、订单、配送记录
- **4 个联系**（菱形）：
  - "属于"（菜品-商家）—— **N:1**（每个菜品属于恰好 1 个商家，1 个商家可有多个菜品）
  - "下单"（用户-订单）—— **1:N**（每个订单属于 1 个用户，1 个用户可有多个订单）
  - "接单"（商家-订单）—— **1:N**（每个订单属于 1 个商家，1 个商家可有多个订单）
  - "配送"（订单-骑手）—— **M:N**（1 个订单可被多位骑手尝试接单，1 个骑手可接多个订单）
- **主码属性下划线**：用户号、商家号、菜品号、骑手号、订单号、（配送表中）订单号+骑手号 组合

**（2）[E-R 转关系模式，3 分]**

```
用户(**用户号**, 姓名, 手机号, 所在宿舍楼或地址)
商家(**商家号**, 商家名, 商家地址, 营业状态)
菜品(**菜品号**, 菜品名, 单价, 商家号*)   -- 商家号 FK → 商家(商家号)
骑手(**骑手号**, 姓名, 手机号, 当前状态)
订单(**订单号**, 用户号*, 商家号*, 下单时间, 期望送达时间, 配送地址, 订单状态, 订单总金额)
       -- 用户号 FK → 用户(用户号)  ON DELETE RESTRICT
       -- 商家号 FK → 商家(商家号)  ON DELETE RESTRICT
配送(**订单号*, **骑手号*, 接单时间, 送达时间)
       -- 订单号 FK → 订单(订单号)
       -- 骑手号 FK → 骑手(骑手号)
```

> 说明：M:N 联系"配送"独立建关系，主码为两端主码的组合 + 自身的非主属性。

**（3）[CREATE TABLE 订单 DDL，2 分]**

```sql
CREATE TABLE 订单 (
    订单号         CHAR(12)        PRIMARY KEY,
    用户号         CHAR(10)        NOT NULL,
    商家号         CHAR(8)         NOT NULL,
    下单时间       DATETIME        NOT NULL,
    期望送达时间   DATETIME        NOT NULL,
    配送地址       VARCHAR(80)     NOT NULL,
    订单状态       VARCHAR(10)     NOT NULL
                                CHECK (订单状态 IN
                                    ('待接单','已接单','配送中','已完成','已取消')),
    订单总金额     DECIMAL(10,2)   NOT NULL
                                CHECK (订单总金额 > 0),
    FOREIGN KEY (用户号) REFERENCES 用户(用户号) ON DELETE RESTRICT,
    FOREIGN KEY (商家号) REFERENCES 商家(商家号) ON DELETE RESTRICT,
    CHECK (期望送达时间 >= 下单时间)
);
```

4 要素逐项核验：
- **实体完整性**：`订单号 CHAR(12) PRIMARY KEY` 【对】
- **参照完整性**：两个外码 + `ON DELETE RESTRICT` 【对】
- **用户定义完整性**：`CHECK (订单总金额 > 0)` + `CHECK (期望送达时间 >= 下单时间)` 【对】
- **状态枚举约束**：`CHECK (订单状态 IN (...))` 【对】

**（4）[触发器，1 分]**

```sql
CREATE TRIGGER trg_rider_free
AFTER UPDATE OF 订单状态 ON 订单
FOR EACH ROW
WHEN (OLD.订单状态 = '配送中' AND NEW.订单状态 = '已完成')
BEGIN
    UPDATE 骑手
       SET 当前状态 = '空闲'
     WHERE 骑手号 IN (
            SELECT 骑手号
              FROM 配送
             WHERE 订单号 = NEW.订单号
           );
END;
```

4 要素逐项核验：
- **触发时机**：`AFTER UPDATE OF 订单状态`（订单状态列更新后）【对】
- **触发条件**：`WHEN (OLD.订单状态 = '配送中' AND NEW.订单状态 = '已完成')`（从'配送中'变'已完成'）【对】
- **触发动作**：`UPDATE 骑手 SET 当前状态 = '空闲'` 【对】
- **关联订单的骑手**：`WHERE 骑手号 IN (SELECT 骑手号 FROM 配送 WHERE 订单号 = NEW.订单号)` 【对】

### 详细解析

- **E-R 图绘制要点**：
  - 矩形（实体） + 椭圆（属性） + 菱形（联系） + 连线
  - 联系类型标注：1:1 / 1:N / M:N 标在连线旁
  - 主码属性下划线
- **E-R 转关系模式三规则**：
  - **1:1 联系**：在任一端加入对方主码作为外码（联系属性也可并入）
  - **1:N 联系**：在多方端加入一方的主码作为外码（联系属性并入多方）
  - **M:N 联系**：**必须独立建一个关系模式**，主码 = 两端主码的组合 + 联系属性
- **本题"配送"是 M:N**：题干"一份订单可被多位骑手尝试接单"是关键——多位骑手对一份订单有"尝试"行为，是 M:N。
- **完整性约束设计**：
  - 实体完整性：PRIMARY KEY 自动保证（主码唯一 + 各分量非空）
  - 参照完整性：FOREIGN KEY + REFERENCES，删除时策略 RESTRICT/CASCADE/SET NULL 三选一
  - 用户定义完整性：CHECK 约束（属性级 / 元组级）、UNIQUE、NOT NULL
- **触发器设计**：
  - 触发时机：BEFORE / AFTER × INSERT / UPDATE / DELETE = 6 选 1 + OF 列名（限定列）
  - 触发动作粒度：FOR EACH ROW（行级）/ FOR EACH STATEMENT（语句级）
  - OLD / NEW 引用：UPDATE 时两者都有（OLD 是更新前，NEW 是更新后）
  - 跨表更新：用子查询关联到订单的骑手
